# P1-b — Issue #3: Tighten entity extractor

```
Issue: https://github.com/shaifulshabuj/docuflow-mcp/issues/3
Branch: v1.6/issue-3-extractor-tightening
[Paste universal preamble above]

## Goal
Eliminate garbage entity extraction. Today the dogfood wiki has 344 entity
pages, many of which are noise (entity_any, entity_add, entity___, etc.)
extracted from markdown bullet lists. Target ≤100 high-signal entity pages
after re-ingest.

## Required reading before editing
Use the Explore sub-agent with this prompt:
    "Survey the entity extraction path in DocuFlow. Starting from
    packages/server/src/tools/ingest-source.ts, trace how entity candidates
    are produced and turned into wiki pages. Identify: (1) where the candidate
    string is decided, (2) where the slug is generated, (3) whether there is
    any existing rejection / dedup logic. Report the function names, file
    paths, and call sequence — under 300 words."

Once you have the survey, proceed.

## Extractor rules to implement (ALL must pass for a candidate to become a page)

1. **Structural anchor required**: candidate must originate from a markdown
   heading (`##`, `###`, `####`), a code block / inline code token, OR a
   link target. Reject candidates pulled from prose bullet lines or
   sentence fragments.

2. **Minimum token signal**:
   - ≥ 2 words, OR
   - ≥ 1 word containing a non-sentence-start capital letter (proper-noun signal:
     `DocuFlow`, `MCP`, `AI`), OR
   - ≥ 1 word with a special token: `_`, `-`, `()`, `::`, `.` (code identifier signal:
     `read_module`, `safeReadFile`, `User.findById`)

3. **Stop-list rejection**: maintain a focused stop-list in a new file
   `packages/server/src/extractor-stoplist.ts`. Start with these terms
   (lowercase; case-insensitive comparison):
     any, add, added, active, audience, behavior, benefits, binary, append,
     approval, audit, auditable, allowed, blocked, the, and, or, for, with,
     from, into, when, then, this, that, these, those, here, there, what,
     why, how, who, where, which, can, will, should, must, may, also,
     just, only, every, each, all, some, one, two, three, true, false,
     yes, no, ok, done, todo
   Expand if the post-extraction sample still shows noise.

4. **No-emoji / no-punctuation-only slugs**: strip emoji from the slug.
   If the resulting slug is empty or matches `/^[_\-]+$/`, reject.

5. **Context requirement**: the entity must have ≥ 1 full sentence (≥ 6 words)
   of surrounding context in the source. Reject if only present as a bare
   bullet item with no prose.

## Implementation steps

1. Add `packages/server/src/extractor-rules.ts` exporting a single
   function `passesEntityRules(candidate: EntityCandidate): { ok: true } | { ok: false, reason: string }`.
   The reason is captured for the rewiki report (#4).

2. Wire the rules into the entity-creation path identified by the Explore
   survey. Reject candidates before they become wiki page writes.

3. Add unit tests in `packages/server/src/__tests__/extractor-rules.test.ts`
   (or matching the existing test layout — check what's already there).
   Cover every rule with at least one positive and one negative case.
   At minimum:
     - heading-extracted candidate passes
     - bullet-list-extracted candidate rejected (rule 1)
     - "any" rejected by stop-list (rule 3)
     - "**" rejected by punctuation-only rule (rule 4)
     - "User.findById" passes (code identifier signal, rule 2)
     - "DocuFlow" passes (proper noun signal, rule 2)
     - bare-bullet candidate without context rejected (rule 5)

4. Document the rules in a new file `docs/EXTRACTOR_RULES.md` so future
   contributors understand why these gates exist and don't relax them.

5. Do NOT re-ingest the existing wiki in this PR. That migration is the
   job of #4 (`docuflow rewiki`). This PR just installs the new gates.

## Acceptance criteria
- [ ] `extractor-rules.ts` and `extractor-stoplist.ts` exist with the 5 rules
- [ ] Unit tests cover each rule (positive + negative); all pass
- [ ] Entity creation path integrates the rules at the right boundary
- [ ] `docs/EXTRACTOR_RULES.md` written
- [ ] Smoke test: manually call ingest_source on a sample markdown with known
      garbage (bullet list of common words) and verify zero new entity pages
      created. Capture command + result in PR body.
- [ ] `npm run build` passes
- [ ] `bash scripts/pre-release-check.sh` prints RESULT: PASSED

## Out of scope
- ML / embedding-based detection (violates moratorium in #1)
- Cross-document entity resolution / dedup (separate concern)
- Re-ingesting the existing `.docuflow/` wiki (that's #4)
- Changing the concept / timeline / synthesis extraction paths
  (only entity extraction is in scope here)

Closes #3.
```

# P2 — Issue #4: `docuflow rewiki` migration command

```
Issue: https://github.com/shaifulshabuj/docuflow-mcp/issues/4
Branch: v1.6/issue-4-rewiki-command
[Paste universal preamble above]

## DEPENDENCY
This issue depends on #2 (synthesiss fix) and #3 (extractor rules) being
merged first. Verify both are on main before starting:
    git log --oneline main | grep -E "(#2|#3)"

If either is missing, stop and report.

## Goal
Add `docuflow rewiki` — a single command that re-ingests all sources with
the new extractor rules, migrates the `synthesiss/` typo (defense-in-depth
even though #2 already cleaned it for the dogfood), backs up the existing
wiki, and produces an audit report. This is how existing users benefit
from #2 + #3 without losing data.

## Required reading
Use the Explore sub-agent:
    "Survey how DocuFlow CLI commands are structured. Look at
    packages/cli/src/commands/init.ts, status.ts, sync.ts, and review.ts.
    Identify: command registration pattern, argument parsing, how a command
    invokes server-side tools, error handling conventions, and quiet/verbose
    flag patterns. Report under 250 words."

## Command behavior

    docuflow rewiki [--dry-run] [--no-backup] [--quiet]

Steps the command executes (in order):

1. **Pre-flight**:
   - Verify `.docuflow/` exists and contains `sources/` and `wiki/`
   - Print summary: source count, current wiki page count by category
   - If --dry-run: skip steps 2 + 4 + 6 (no writes), still produce the report

2. **Backup**:
   - Default ON. Skipped only if `--no-backup` is passed.
   - Copy `.docuflow/wiki/` → `.docuflow/wiki.backup-<ISO-timestamp>/`
   - Use `cp -r` via Waymark bash. Print backup path.

3. **Migrate synthesiss/** (defense-in-depth for existing users):
   - If `.docuflow/wiki/synthesiss/` exists, move its contents to `syntheses/`
   - Remove the empty `synthesiss/` directory
   - Log every moved file

4. **Re-ingest all sources**:
   - List every file in `.docuflow/sources/`
   - For each, call the existing `ingest_source` tool handler programmatically
     (do NOT shell out to the MCP server; import the handler directly from
     `@doquflow/server`)
   - The new extractor rules from #3 are now active, so noise gets filtered out
   - User-saved synthesis pages (those produced by `save_answer_as_page`,
     identifiable by the `saved_from_query: true` frontmatter field — confirm
     this field exists during the Explore survey; if not, look for a different
     signal in the page metadata) are PRESERVED unchanged

5. **Update index**:
   - Call `update_index` handler so `index.md` reflects the new state

6. **Write report** to `.docuflow/rewiki-report.md`:
   - Timestamp, dry-run flag, backup path (or "skipped")
   - Pages before / after by category
   - List of removed pages with rejection reason (pulled from
     extractor-rules.ts return values)
   - List of preserved user-synthesized pages
   - List of synthesiss/ migrations (if any)

7. **Append to log.md** a one-line entry: `rewiki: removed N, kept M, preserved K user pages`

## CLI integration
- Add `packages/cli/src/commands/rewiki.ts` following the pattern from #3's Explore output
- Register the command in `packages/cli/src/index.ts`
- Update `docuflow --help` output (or whatever generates it) to list rewiki
- Update CLAUDE.md generation in init.ts to mention rewiki under "Maintenance"

## Acceptance criteria
- [ ] `docuflow rewiki --dry-run` runs in under 5 seconds on this repo and
      outputs a report without modifying anything
- [ ] `docuflow rewiki` (real run) reduces this repo's entity page count
      from 344 to ≤100 (proves #3 is wired correctly through the migration)
- [ ] Backup directory created at `.docuflow/wiki.backup-<timestamp>/`
- [ ] User-saved synthesis pages preserved unchanged (verify by content hash
      of at least one such page before/after)
- [ ] `.docuflow/rewiki-report.md` written with rejection reasons grouped
- [ ] If interrupted by Ctrl-C mid-run, the backup is intact (basic safety)
- [ ] Unit tests cover: dry-run produces no writes, backup created when
      not disabled, report fields populated correctly
- [ ] README.md updated: add `docuflow rewiki` under maintenance commands
- [ ] CHANGELOG.md + release/CHANGELOG.md entry under [Unreleased] →
      "feat: add rewiki migration command (#4)" — both mirrored exactly
- [ ] `npm run build` passes
- [ ] `bash scripts/pre-release-check.sh` prints RESULT: PASSED

## Out of scope
- Cron / auto-rewiki scheduling (re-introduces the daemon pattern this
  whole reset is moving away from)
- Per-source granular re-ingest (bulk only — keeps CLI surface minimal)
- Telemetry / phone-home about rewiki results

Closes #4.
```
